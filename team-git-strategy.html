Purpose & Scope

Applies to: All developers working on this repository.

1. Branching Model

Our team follows a feature-branch workflow with protected main branches.

Primary Branches

BranchPurposeNotes
mainProduction-ready codeProtected (no direct push)
develop (optional)Staging / pre-release codeProtected (no direct push)

Supporting Branches

Hotfixhotfix/prod-crashImmediate production fixes

Rules

  • Developers must not work directly on main.
  • Every task must have its own feature branch.
  • Branch names must be meaningful and hyphenated, e.g. feature/module-description, bugfix/cart-discount-issue.

2. Merge Strategy

We use a merge-first workflow for shared branches.

✔ Allowed merge methods

  • Merge Commit (default)
  • Squash and Merge for messy feature branches

❌ Not allowed

  • “Rebase and Merge” into main
  • Force-pushing into shared branches

3. Rebase Strategy

Rebasing is allowed only in specific situations.

✔ You MAY use rebase:

  • On your own feature branch
  • Before raising a Pull Request
  • To tidy commits (git rebase -i)

❌ You MUST NOT use rebase:

  • On any shared/pushed team branch
  • On main, develop, or release branches
  • After others already pulled your changes

Rule of thumb: Rebase your work; never rebase others’ work.

4. Pull Request / Merge Request Workflow

  1. Update your feature branch 
    git fetch origin
git rebase origin/main   # or merge if you prefer
  2. Push updates

    If rebased earlier, safety push:

    git push --force-with-lease
  3. Create a Pull Request

    Checklist:

    • Clear title (what the feature is)
    • Description (what changed + screenshots if UI)
    • Unit tests (if applicable)
    • No debug code / console logs
    • CI pipeline must pass
  4. Review Process

    Minimum 1–2 peer approvals required. Reviewers check:

    • Code quality
    • Naming convention
    • Architectural alignment
    • Breaking changes
    • Security concerns
  5. Merge safely into main

    Prefer Merge Commit. Use Squash and Merge for multiple small commits.

  6. Delete merged branch

    Keeps the repo clean.

5. Conflict Resolution Guidelines

When merging:

  • Resolve conflicts once
  • Add all resolved files
  • Commit merge

When rebasing:

  • Resolve conflicts for each commit
  • Continue:
git add .
git rebase --continue

Best practice: Resolve conflicts using local editor, not GitHub/GitLab UI.

6. Commit Message Guidelines

We follow Conventional Commit Style:

Format:

type(scope): short summary

Examples:

  • feat(auth): add jwt token refresh support
  • fix(order): handle null order id
  • refactor(ui): optimize component structure
  • docs(api): update swagger docs

Accepted types: feat, fix, refactor, perf, test, docs, style, chore.

7. Branch Protection Rules

Protected branches: main, develop (if used).

Protection includes:

  • No direct pushes
  • PR required
  • Minimum 1–2 reviewer approvals
  • All CI checks must pass
  • No force push
  • Require up-to-date with target branch before merging

8. Release Management

Release branch approach:

git checkout -b release/v1.5.0
  • Fix minor issues
  • Write release notes
  • Merge into:
  • main → release version
  • develop → sync changes

Tagging:

git tag -a v1.5.0 -m "Release 1.5.0"
git push origin v1.5.0

10. Do & Don’t List

✔ DO

  • Use feature branches for all tasks
  • Update your branch regularly
  • Use rebasing only before PR submission
  • Keep commits small and meaningful
  • Delete merged branches
  • Follow Conventional Commits

❌ DON’T

  • Rebase shared branches
  • Force-push to main / develop
  • Keep long-running feature branches without updates
  • Push code without testing
  • Merge failing pipelines

11. Team Responsibility

Every team member is responsible for:

  • Following Git standards
  • Keeping histories clean
  • Maintaining branch discipline
  • Conducting proper code reviews
  • Avoiding unsafe operations
